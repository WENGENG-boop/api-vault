# API Vault

中文说明: [README.zh-CN.md](README.zh-CN.md)

API Vault is a local browser dashboard for managing multiple API providers, API keys, proxy base URLs, usage records, and provider-reported usage data.

It is designed for people who use many official or third-party AI API platforms and want one place to see:

- Which providers and API keys are configured
- Which model was called
- How many requests were made
- How many tokens were used
- Which calls succeeded or failed
- What the provider returned for balance or usage data
- Which local proxy base URL should be copied into another app

The project runs locally by default. Users add their own API keys after starting the app. This repository does not include any API keys or private vault data.

## Core Idea

Many AI tools allow users to configure a custom API base URL and API key. API Vault sits between those tools and the real upstream provider.

Instead of putting the provider's original base URL directly into another app, you copy the **API Vault Base URL**:

```text
http://127.0.0.1:3210/proxy/<providerId>/v1
```

Then the request flow becomes:

```text
Third-party app
  -> API Vault local proxy
  -> Real upstream provider
  -> API Vault records the call
  -> Response is returned to the third-party app
```

API Vault injects the real API key, forwards the request, records the result, and displays usage in the dashboard.

## API Vault Base URL

The most important field in this app is:

```text
API Vault Base URL - copy this into the third-party app
```

This is the URL that lets API Vault record usage.

When you add a provider and API key, API Vault shows two different URLs:

```text
Original Base URL
```

This is the real upstream provider URL, for example:

```text
https://api.openai.com/v1
https://openrouter.ai/api/v1
https://api.deepseek.com/v1
```

and:

```text
API Vault Base URL
```

This is the provider-level local proxy URL generated by API Vault. All keys under the same provider share this URL, for example:

```text
http://127.0.0.1:3210/proxy/provider_abc/v1
```

You should copy the **API Vault Base URL**, not the Original Base URL, into the third-party app.

### Where To Paste It

In the third-party app, find the setting usually called one of these:

- Base URL
- API Base URL
- Custom API URL
- OpenAI Compatible Base URL
- Endpoint

Paste the **API Vault Base URL** there.

Example:

```text
Third-party app Base URL:
http://127.0.0.1:3210/proxy/provider_abc/v1
```

The API key field in the third-party app should stay as the user's own real API key. API Vault uses that key to decide which key under the provider should receive the usage record.

### Why This Is Required

API Vault can only record calls that pass through API Vault.

If a third-party app calls this directly:

```text
https://api.openai.com/v1
```

API Vault will not see the request, so no usage record will appear.

If the third-party app calls this instead:

```text
http://127.0.0.1:3210/proxy/provider_abc/v1
```

API Vault can:

- Forward the request to the real provider
- Inject or match the correct API key
- Record the model name
- Record status, tokens, latency, and errors
- Show the call in the dashboard and usage page

### Local App vs Cloud App

If the third-party app runs on the same computer as API Vault, use:

```text
http://127.0.0.1:3210
```

or:

```text
http://localhost:3210
```

If the third-party app is a cloud website or cloud server, it cannot access your personal computer's `127.0.0.1`.

For cloud usage, deploy API Vault on a server and use a public HTTPS URL, for example:

```text
https://api-vault.example.com/proxy/provider_abc/v1
```

Then paste that HTTPS API Vault Base URL into the cloud platform.

## What Gets Tracked

For each API call that goes through the API Vault proxy, the app records:

- Time
- Provider
- Base URL
- API key name or masked key
- Model
- Request status
- Input tokens
- Output tokens
- Total tokens
- Cost, if returned or available
- Latency
- Error message, if the call failed

API Vault can only track requests that pass through its proxy URL. If another app calls the upstream provider directly, API Vault cannot see that request.

## Provider And API Key Model

API Vault groups API keys by provider.

For example, these keys:

```text
https://api.openai.com/v1  key1
https://api.openai.com/v1  key2
https://api.openai.com/v1  key3
```

are shown as one provider:

```text
OpenAI
  - key1
  - key2
  - key3
```

The provider shows combined statistics. Each key also shows its own statistics.

If a new key is added with a base URL that already exists, API Vault adds it under the existing provider instead of creating a duplicate provider.

## First-Time Setup

1. Open `http://127.0.0.1:3210`.
2. Set a master password.
3. Add a provider or API key.
4. Enter the provider's original base URL, such as:

```text
https://api.openai.com/v1
https://api.anthropic.com
https://openrouter.ai/api/v1
https://api.deepseek.com/v1
```

5. Enter your own API key.
6. Copy the API Vault Base URL shown in the provider card.
7. Paste that API Vault Base URL into the third-party app's Base URL field.
8. Use the same API key in the third-party app if it requires one.

After that, calls made by the third-party app should appear in API Vault.

## Important Proxy Notes

For local desktop apps running on the same computer, use:

```text
http://127.0.0.1:3210
```

or:

```text
http://localhost:3210
```

For cloud platforms, `127.0.0.1` will not work because it points to the cloud platform's own machine, not your computer.

If a cloud service needs to call API Vault, deploy API Vault on a reachable server and put it behind HTTPS.

## Data Storage

Local data is created only after the user starts the app.

By default, data is stored in:

```text
.api-vault/vault.json
```

This file contains encrypted provider and API key data plus local usage records.

The clean repository intentionally excludes:

```text
.api-vault/
node_modules/
dist/
dist-main/
*.log
.env
```

## Privacy And Security

- API keys are not included in this repository.
- Users enter their own API keys after running the app.
- API keys are stored locally.
- The vault file is excluded from Git by `.gitignore`.
- API keys are masked in the UI.
- Full keys are only decrypted when needed for proxying or copying.

This project is intended as a local self-hosted tool. Do not expose it publicly without authentication, HTTPS, firewall rules, and a clear deployment plan.

## Docker Data Persistence

The Docker Compose file maps local data into the container:

```yaml
volumes:
  - ./.api-vault:/app/.api-vault
```

This means the user's vault data stays in the project folder on their own machine and survives container restarts.

To reset local data:

```bash
docker compose down
rm -rf .api-vault
```

On Windows PowerShell:

```powershell
docker compose down
Remove-Item -Recurse -Force .\.api-vault
```

## Balance And Usage Sync

Different third-party providers expose balance and usage data in different formats.

API Vault supports custom balance sync rules:

- Balance URL
- HTTP method
- Headers
- JSON path for extracting returned values

If a provider returns usage as `token_usage`, `tokens`, or another unit, API Vault displays the returned unit instead of forcing a currency conversion.

## Remote Public Proxy

For remote computers, CI, or a public tunnel, do not expose real provider API keys. Create a `proxy_xxx` token in the **Proxy Tokens** page, map public model names to internal providers, then call:

```text
POST /proxy/v1/chat/completions
GET  /proxy/v1/models
```

Use:

```text
Authorization: Bearer proxy_xxx
```

API Vault validates the proxy token, applies model/provider permissions and rate limits, injects the real encrypted provider key internally, forwards the request, and records usage. See [REMOTE_ACCESS.md](./REMOTE_ACCESS.md) for Cloudflare Tunnel, Tailscale, and reverse proxy deployment notes.
